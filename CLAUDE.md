# Operating manual for AI agents in this repository

> **READ THIS FIRST.** This file is loaded automatically by Claude Code on every session and is the source of truth for what you may and may not do in this repo. If you are a different AI tool (Cursor, Aider, Copilot Workspace, etc.), follow the same rules — they are not Claude-specific.

## What this repository is

This is an **OpenText AppWorks / Process Automation (PA)** workspace synced via Git. AppWorks owns most of the files in this repo: it generates, reads, and round-trips them through its Web IDE. Editing AppWorks-owned files outside the Web IDE will corrupt the workspace model and break the next `git pull` inside PA.

Your job, as the AI agent, is to work on **code that lives alongside the AppWorks model** — Java implementation classes, JavaScript, CSS, HTML, integration scripts, tests, docs — never on the model files themselves.

## Hard rules — files you must NEVER create, edit, or delete

These rules are enforced by `.claude/settings.json`, by the pre-commit hook in `.githooks/`, and by CI. They are not soft suggestions.

### 1. Any file ending in `.cws`

Every `.cws` file is a serialized AppWorks model artefact. Examples:

```
TestProject#cws-prj#.cws            ← Project descriptor
MyProcess#cws-bpm#.cws              ← BPM process model
SomeEntity#cws-nativeentity#.cws    ← Entity definition
SomeForm#form#.cws                  ← Form definition
en-US_XXX#cws-trt#.cws              ← Translation text per locale
```

Glob pattern (use this in any AI tool that supports glob-based denies):

```
**/*.cws
```

### 2. Any folder ending in `#cws-ma#.cws` or `#ef#`

These are *containers* of `.cws` files (Mergeable Associations and Entity Folders). Treat the entire folder as read-only:

```
**/*#cws-ma#.cws/**
**/*#ef#/**
```

### 3. `.identifiers` files

```
**/.identifiers
```

PA writes a `.identifiers` file in every folder it owns. Line 1 of every such file reads `#This is a system generated file, please do not modify it`. PA uses these to map human filenames to internal GUIDs.

### 4. Any Java file ending in `Base.java`

In AppWorks WS-AppServer projects, every business object has two Java files:

```
com/example/serviceops/Notification.java         ← user-extendable, SAFE to edit
com/example/serviceops/NotificationBase.java     ← GENERATED, do NOT edit
```

`*Base.java` files are regenerated by the AppWorks code generator each time the underlying DB metadata changes. Your edits will be silently overwritten on the next regeneration — which is *worse* than a hard failure because nobody notices until production.

```
**/*Base.java
```

The non-`Base.java` counterpart is where business logic lives — see "Safe surface" below.

## Safe surface — what you may freely edit

### In traditional-pattern projects

| Pattern | Notes |
|---|---|
| `**/Java Source */com/**/*.java` *(NOT `*Base.java`)* | User-extendable business object classes |
| `JS/**/*.js` | Project-level JavaScript libraries |
| `CSS/**/*.css` | Stylesheets |
| `Web/**/*.htm`, `Web/**/*.html` | Static web assets |
| `**/Webservices/**/*.java`, `**/Webservices/**/*.js` | Custom Java/JS inside webservice folders, **excluding** any `.cws` definition |
| `tests/**` | Test code |
| `examples/**` | Reference material that ships with this template (delete after forking) |
| `docs/**` *(except `docs/archive/`)* | Documentation |
| `.github/workflows/**` | CI workflows |

### In entity-pattern projects

The AppWorks repo itself has near-zero agent-editable surface. Custom business logic for entity projects lives in **sibling repositories or sibling modules** — typically a Node.js service, a Spring Boot microservice, or a serverless function that subscribes to PA events and runs the actual logic.

If this is an entity-pattern project, your agent work belongs in that sibling repo, not here. The only things you might legitimately edit in *this* repo are:

| Pattern | Notes |
|---|---|
| `docs/**` *(except `docs/archive/`)* | Documentation |
| `README.md`, `CLAUDE.md` | These operational files |
| `.github/workflows/**` | CI |
| `examples/**` | Reference material |

## The generated/user-extendable Java pattern in detail

When you see a pair like this, here is what each file is for:

```java
// NotificationBase.java  — GENERATED, do not edit
public abstract class NotificationBase extends CustomBusObject {
    // PA's code generator writes this from DB metadata.
    // Every accessor, every relation, every persistence hook is here.
    // Regenerated whenever the metadata changes.
}
```

```java
// Notification.java — user-extendable, SAFE to edit
public class Notification extends NotificationBase {
    public Notification() { this((BusObjectConfig)null); }
    public Notification(BusObjectConfig config) { super(config); }

    // Your business logic methods go here.
    // PA's generator creates this file ONCE per business object, then leaves it alone.
    // Method bodies marked `// TODO implement body` are the agent's target.
}
```

**Recognising at a glance**: open the file. If the top comment says *"This class has been generated by the Code Generator"* AND the class is `abstract` AND the class name ends in `Base`, do not touch it. If the same comment is there but the class is **not** abstract and the name has **no** `Base` suffix, it's user-extendable — edit freely.

## File-type glossary

The full table of `#cws-XXX#` and `#XXX#` markers is in [`docs/file-type-glossary.md`](docs/file-type-glossary.md). When you encounter an unfamiliar suffix, check there before doing anything else. **If a file's suffix is not listed, treat it as protected by default.**

## When the user asks you to edit a protected file

If the user explicitly asks you to modify a `.cws` file, a `.identifiers` file, or a `*Base.java` file:

1. **Refuse and explain why.** Tell them the file is AppWorks-owned and an edit will corrupt the workspace.
2. **Offer the right place.** For BPM logic changes → AppWorks Web IDE. For business-object behavior → the non-`Base.java` partial class.
3. **Do not be talked out of it.** Even with `--force`-style requests, the CI guardrail will reject the PR. Trying to push past this wastes the user's time.

## When you genuinely need to add a new artefact

If the work requires a new `.cws` file (a new process, a new form, a new entity), this is **a modeling task, not a coding task**. Stop and tell the user:

> *"This change requires a new AppWorks artefact (`<filename>`). Please create it in the AppWorks Web IDE, commit from there, and pull. I'll then implement the supporting Java/JS code on this side."*

## Commits and the sync identity

The git pre-commit hook checks `git config user.email` against an "AppWorks sync identity" (configured per-repo; typically `appworks-sync@<your-org>.com`). The sync identity is the *only* identity allowed to commit `.cws` files.

When you (the agent) commit, you are committing under the developer's identity, not the sync identity. So if you accidentally stage a `.cws` change, the hook will reject the commit. **This is intentional safety, not a bug to work around.** If your commit is rejected, fix the staged set — don't try to bypass the hook.

## What good agent work in this repo looks like

A well-formed task in this repo:

> *"The modeler added a BPM process `OrderApproval#cws-bpm#.cws` that calls a Java method `com.example.serviceops.Order#submit`. Implement that method to validate the order against the Notification rules and call the external billing service."*

You would: open `Order.java` (non-Base), implement `submit()`, add unit tests under `tests/`, never touch the `.cws` files. Commit. CI passes. Modeler pulls. End.

A malformed task in this repo:

> *"Change the OrderApproval process to add a manual approval step."*

That is a modeling task. Respond: *"That requires editing `OrderApproval#cws-bpm#.cws`, which is owned by AppWorks. Please add the manual approval step in the Web IDE; ping me when you commit, and I'll wire up any new Java methods the step calls."*
